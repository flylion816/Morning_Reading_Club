# 设计：我发起的请求看见状态

## 背景
当前小程序的请求 UI 主要面向“收到的请求”。`InsightRequest` 模型已经支持 `pending`、`approved`、`rejected`、`revoked` 四种状态，后端也已经提供 `GET /insights/requests/sent`。因此本次重点是补齐前端体验，而不是新增数据模型。

## 目标
- 用户能在首页预览和完整请求页查看自己发起的请求状态。
- 保留现有收到请求的同意、拒绝能力。
- 用清晰的状态标签降低扫描成本。
- 首页卡片在小屏手机上保持紧凑、美观，不挤压内容。

## 非目标
- 不执行数据库重置、初始化或种子脚本。
- 不新增请求状态值。
- 不调整管理后台流程。
- 不调整付费权限规则。

## 交互设计
- 使用两个分段切换项：`收到的` 和 `我发起的`。
- 默认展示 `收到的`，保持现有工作流不变。
- 状态文案按视角区分：
  - 收到的请求：`待处理`、`已同意`、`已拒绝`、`已撤销`
  - 我发起的请求：`等待中`、`已同意`、`已拒绝`、`已撤销`
- 我发起的请求为只读状态，不展示同意/拒绝按钮。
- 已同意的发起请求在有上下文时可跳转到对方主页并携带期次信息；等待中和已拒绝记录主要用于查看状态。
- 空状态按方向区分：
  - 收到的：`暂无收到的请求`
  - 我发起的：`你还没有发起过看见请求`

## 视觉方向
- 分段切换放在区块标题下方，紧贴列表。
- 卡片圆角和间距遵循现有首页卡片风格。
- 状态标签使用柔和底色：等待/待处理为蓝色，已同意为绿色，已拒绝为红色，已撤销为灰色。
- 避免大段说明文案，每张卡片只保留用户行、元信息行和时间行。
- 长昵称、期次名、标题必须截断，不能挤压状态标签或按钮。

## 数据流
- `收到的` 调用 `insightService.getReceivedRequests()`。
- `我发起的` 调用 `insightService.getSentRequests()`。
- 格式化时按视角选择展示用户：
  - 收到的请求展示 `fromUserId`。
  - 我发起的请求展示 `toUserId`。
- 完整请求页可以并行加载两个列表；首页预览只展示当前方向的前几条，保持页面紧凑。
