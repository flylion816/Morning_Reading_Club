## 发现的问题

### P0/P1 需要修复

1. 管理后台 `vue-tsc --build` 失败。
   - `admin/vitest.config.ts` 使用 Vitest 4 不支持的 `coverage.provider: 'c8'`。
   - `tsconfig.app.json` 把 `src/tests/**`、`*.test.ts` 等测试文件纳入生产类型检查。
   - `admin/src/services/api.ts` 的响应拦截器实际返回 `data`，但 TypeScript 仍推断 API 方法返回 `AxiosResponse`，导致大量 store/view 类型误报。
   - `periodApi` 缺少 `getPeriodById`，但 `period` store 调用了它。
   - `userApi.getUsers` 只接收 params 对象，但 `user` store 使用 `(page, pageSize)` 调用。

2. 管理后台 insight request API 路径不一致。
   - `insightRequestsApi` 指向 `/admin/requests`，后端实际挂载在 `/api/v1/insights/admin/requests`。
   - 页面 `InsightRequestsManagementView.vue` 使用的是正确路径，service 导出路径错误，后续调用会 404。

3. 后端 `insight.routes.js` 存在重复 `PUT /:insightId`。
   - 用户更新路由先注册 `authMiddleware`，管理员更新路由后注册 `adminAuthMiddleware`，后者不可达。
   - 管理后台 `insightApi.updateInsight()` 使用 `/insights/:id`，会被用户鉴权路由截获，管理员 token 场景可能失败。

4. 管理后台 Vitest 测试不稳定/失败。
   - `RichTextEditor.spec.ts` 仍按旧 textarea/markdown 编辑器断言，当前组件已切换为 Quill。
   - `stats.spec.ts` 直接调用真实 axios 请求，没有 mock/await，产生未处理网络错误。

5. 后端测试不稳定/失败。
   - 多个集成测试各自创建 MongoMemoryServer 并重复连接 mongoose，和全局测试 setup 冲突。
   - `insight-request-flow.integration.test.js` 的 `clone()` 会复制函数，触发 `structuredClone` 异常，导致 500。
   - `period-section.integration.test.js` 的期次测试仍按旧静态状态和默认公开数据假设断言，和当前“只返回已发布期次 + 动态状态计算”逻辑不一致。
   - `checkin.controller.test.js` 的 populate mock 链未覆盖当前控制器的四次 populate。
   - `ranking.controller.test.js` 仍期望空数据时 `currentUser` 为 `null`，当前实现返回当前用户的零打卡状态对象。
   - `study-reminder.service.test.js` 使用 `Date.now()` 构造发送日期，当前日期超过测试期次后变成过期，导致定时提醒测试失败。

### P2 后续质量债

6. 管理后台和小程序存在大量调试 `console.log`，包括 API interceptor、router guard、多个小程序页面。
7. 后端 `server.js` 启动时有临时 `[DEBUG] After dotenv loading` 控制台输出。
8. 根目录 lint 脚本用 `|| true` 吞掉失败，质量门禁不可依赖。已修复为真实后端 lint 门禁。
9. 后端 ESLint 风格债很多。已完成策略调整：正确性规则保持阻塞，历史风格债降为 warning，避免把功能修复和大规模风格重构混在一起。

## 修复策略

- 先修会导致真实功能失败或质量门禁失败的问题：管理后台 type-check、API 路径、管理员 insight 更新路由、红灯测试。
- 后端测试优先修测试夹具与当前契约不一致的问题；不执行任何数据库初始化或重置脚本。
- 不做大规模 ESLint 风格重构；改为收紧质量门禁策略，让真实正确性问题阻塞，历史风格问题保留为后续可分批清理的 warning。
