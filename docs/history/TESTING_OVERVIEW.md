## 📊 测试金字塔详解

```
                         E2E Tests (5%)
                        ┌──────────────┐
                        │ Cypress      │ ← 核心用户流程
                        │ E2E Tests    │   (登录、打卡、管理)
                        └──────────────┘
                   Integration Tests (25%)
              ┌────────────────────────────┐
              │ Backend API Integration    │ ← API完整流程
              │ Database Interactions      │   (注册→登录→打卡)
              └────────────────────────────┘
         Unit Tests (70%)
┌───────────────────────────────────────────────┐
│ Controllers | Models | Utils | Components    │ ← 独立功能测试
│ Services | Middleware | Store | Composables  │   (函数、方法、类)
└───────────────────────────────────────────────┘
```

### 各层级说明

| 测试层级        | 覆盖范围      | 执行速度      | 维护成本 | 反馈质量   |
| --------------- | ------------- | ------------- | -------- | ---------- |
| **E2E**         | 完整用户流程  | 慢（5-10min） | 高       | 最接近真实 |
| **Integration** | API+数据库    | 中（2-5min）  | 中       | 真实度高   |
| **Unit**        | 单个函数/组件 | 快（<2min）   | 低       | 定位精准   |

## 🎓 学习路径

### 新手开发者

1. **第1周**: 阅读 `TESTING_QUICKSTART.md`
   - 安装依赖
   - 运行现有测试
   - 理解测试结构

2. **第2周**: 编写第一个单元测试
   - 选择简单的工具函数
   - 编写测试用例
   - 运行并通过测试

3. **第3周**: 编写集成测试
   - 理解API测试
   - 使用Supertest
   - 测试完整流程

4. **第4周**: 熟悉CI/CD
   - 查看GitHub Actions
   - 理解失败原因
   - 修复CI问题

### 资深开发者

1. 快速浏览 `TESTING_STRATEGY.md`
2. 配置Git Hooks和CI/CD
3. 编写E2E测试
4. 优化测试性能
5. 指导新人

## 💡 最佳实践

### ✅ Do

- 先写测试再写代码（TDD）
- 测试命名清晰描述意图
- 每个测试只验证一件事
- 使用AAA模式（Arrange-Act-Assert）
- Mock外部依赖
- 保持测试独立性

### ❌ Don't

- 测试实现细节
- 依赖测试执行顺序
- 在测试中使用setTimeout
- 忽略失败的测试
- 过度Mock导致测试无意义
- 提交带有console.log的代码
